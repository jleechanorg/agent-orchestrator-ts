Agent Gemini Plugin

Design reference for @composio/ao-plugin-agent-gemini — integrates Gemini CLI into the AO plugin system as a first-class agent backend.

process: gemini session path: ~/.gemini/tmp/<sha256>/chats/ runtime: tmux permission flag: --yolo auth: OAuth (oauth-personal)

Purpose

Wraps the gemini CLI so AO can spawn, supervise, resume, and introspect Gemini CLI sessions the same way it manages Claude Code and Cursor sessions. No AO core changes required — the plugin implements the standard Agent interface from @composio/ao-core.

Relationship to agent-claude-code

Both plugins are thin wrappers around @composio/ao-plugin-agent-base, which provides the full Agent implementation via createAgentPlugin(config, overrides?). The two differ only in config:

  • Process name: gemini vs claude
  • Session path: ~/.gemini/tmp/<sha256>/chats/ vs ~/.claude/projects/
  • Session encoding: SHA-256 hash vs path-mangling
  • Session file extension: .json vs .jsonl
  • Permission flag: --yolo vs --dangerously-skip-permissions
  • System prompt: GEMINI_SYSTEM_MD env var (no CLI flag) vs --append-system-prompt

Shared implementation lives in packages/plugins/agent-base.

Launch

  • gemini — base command
  • -r / --resume <sessionId> — resume existing session
  • --yolo — auto-approval / permissionless mode
  • --model <name> — optional model override
  • Prompt delivered post-launch via sendMessage()
  • System context injected via GEMINI_SYSTEM_MD env var (no CLI flag)

Session Tracking

  • JSON files in ~/.gemini/tmp/<sha256(workspacePath)>/chats/
  • SHA-256 path encoding (distinct from Claude Code's path-mangling)
  • Latest file selected by mtime
  • Tail-parse for summary, cost, and session ID
  • toGeminiProjectPath() computes SHA-256 of normalized workspace path

Activity Detection

  • Idle: no JSON writes in last 8 s
  • Working: recent JSON write activity
  • Stuck: no new JSON writes beyond stuck threshold
  • Process check via ps across all tmux pane TTYs

Authentication

Gemini CLI supports two auth modes. AO uses OAuth (oauth-personal) — the same file-based approach as Claude Code (~/.claude/) and Cursor (~/.cursor/). No env vars needed at runtime.

ModeSettingCredentials
OAuth (recommended) selectedType: "oauth-personal" ~/.gemini/oauth_creds.json (Google Bearer token)
API key selectedType: "gemini-api-key" GEMINI_API_KEY env var

Switch via ~/.gemini/settings.json → security.auth.selectedType. OAuth shows "Loaded cached credentials." on startup; API key mode requires the env var in every shell context.

Agent Interface Implementation

  • getLaunchCommand(config) — builds the gemini CLI string
  • getEnvironment(config) — injects AO_SESSION_ID, metadata env vars, and GEMINI_SYSTEM_MD
  • isProcessRunning(handle) — checks tmux pane TTYs via ps for gemini
  • getActivityState(handle) — JSON mtime-based idle/working/stuck classification
  • getSessionInfo(session) — extracts summary, cost, and session UUID from JSON tail
  • getWorkspaceHooks() — installs metadata-updater hook

Plugin Registration

Registered as a built-in plugin in the AO plugin loader:

// packages/plugins/dist/index.js
import geminiPlugin from "@composio/ao-plugin-agent-gemini";

export const builtinPlugins = [
  claudeCodePlugin,
  codexPlugin,
  geminiPlugin,   // ← added
  ...
];

Consumers can also override it per-project via ao.config.json agentPlugin key.

Constraints & Known Gaps

  • One-shot vs interactive: gemini -p (headless) exits after one response. Interactive mode is required for multi-turn AO sessions.
  • Cost estimation accuracy: Gemini 2.0 Flash pricing ($0.10/M input, $0.40/M output) used as baseline. Will be inaccurate for other models until Gemini CLI exposes per-call cost in JSON output.
  • No system prompt flag: No --append-system-prompt equivalent. System context delivered via GEMINI_SYSTEM_MD env var, injected by getEnvironment().

Testing
