Agent Cursor Plugin

Design reference for @composio/ao-plugin-agent-cursor — integrates Cursor Agent CLI into the AO plugin system as a first-class agent backend.

process: cursor-agent session path: ~/.cursor/projects/ runtime: tmux permission flag: --force

Purpose

Wraps the cursor-agent CLI so AO can spawn, supervise, resume, and introspect Cursor Agent sessions the same way it manages Claude Code and Codex sessions. No AO core changes required — the plugin implements the standard Agent interface from @composio/ao-core.

Relationship to agent-claude-code

Both plugins are thin wrappers around @composio/ao-plugin-agent-base, which provides the full Agent implementation via createAgentPlugin(config, overrides?). The two differ only in config:

  • Process name: cursor-agent vs claude
  • Session directory: ~/.cursor/projects/ vs ~/.claude/projects/
  • Permission flag: --force vs --dangerously-skip-permissions
  • System prompt: not supported (undefined) vs --append-system-prompt

Shared implementation lives in packages/plugins/agent-base.

Launch

  • cursor-agent — base command
  • --force — permissionless / auto-edit mode
  • --model <name> — optional model override
  • Prompt delivered post-launch via sendMessage()

Session Tracking

  • JSONL files in ~/.cursor/projects/<encoded-path>/
  • Latest file selected by mtime
  • Tail-parse for summary, cost, and session ID
  • toCursorProjectPath() encodes workspace path identically to how Cursor CLI does

Activity Detection

  • Idle: no JSONL writes in last 8 s
  • Working: recent JSONL write activity
  • Stuck: no new JSONL writes beyond the stuck threshold
  • Process check via ps across all tmux pane TTYs

Agent Interface Implementation

  • getLaunchCommand(config) — builds the cursor-agent CLI string
  • getEnvironment(config) — injects AO_SESSION_ID and metadata env vars
  • isProcessRunning(handle) — checks tmux pane TTYs via ps for cursor-agent
  • getActivityState(handle) — JSONL mtime-based idle/working/stuck classification
  • getSessionInfo(session) — extracts summary, cost, and session UUID from JSONL tail
  • getWorkspaceHooks() — installs .cursor/metadata-updater.sh hook

Plugin Registration

Registered in the core built-in plugin registry:

// packages/core/src/plugin-registry.ts
{ slot: "agent", name: "cursor",
  pkg: "@composio/ao-plugin-agent-cursor" },

Consumers can also override it per-project via ao.config.json agentPlugin key.

Differences from Claude Code

  • Permission flag: --force vs --dangerously-skip-permissions.
  • No --append-system-prompt: Cursor Agent CLI does not support this flag; system context must be included in the first sendMessage() call post-launch.
  • Session restore: Claude Code uses JSONL-derived UUIDs with --resume; Cursor requires SQLite lookup (unimplemented, returns null).
  • JSONL schema: Cursor session files follow a similar envelope structure but use Cursor-specific event types for tool calls.

Constraints & Known Gaps

  • Session restore unimplemented: getRestoreCommand returns null. Cursor stores sessions in SQLite at ~/.cursor/chats/, not JSONL. Resume requires SQLite introspection (chat ID lookup) which is not yet implemented.
  • No system prompt at launch: Cursor Agent CLI has no equivalent of --append-system-prompt. System context must be woven into the initial sendMessage() call post-launch.
  • Cost estimation: Uses Claude Sonnet pricing as a placeholder ($3/M input, $15/M output). Cursor does not expose per-token costs in its JSONL output.

Testing

  • src/index.test.ts — manifest, launch command, process detection, session parsing (68 tests)
  • src/__tests__/activity-detection.test.ts — JSONL mtime-based activity states with real temp directories (42 tests)
  • All 110 tests pass; no mocks of the filesystem in activity tests (real fs operations)
