514o · Auto-merge wiring

Before

case "auto-merge": {
  // For now, just notify
  await notifyHuman(event, "action");
  return { action: "auto-merge", success: true };
}

scm.mergePR() existed in scm-github/src/index.ts:631 but was never called from lifecycle.

After

case "auto-merge":
case "request-merge": {
  const mergeMethod = reactionConfig.mergeMethod ?? "squash";
  if (reactionKey === "request-merge") {
    await notifyHuman(approvalEvent, "action");
  }
  await scm.mergePR(session.pr, mergeMethod);
  return { action, mergeMethod, success: true };
}
  • auto-merge — merges directly when session reaches mergeable
  • request-merge — notifies human first, then merges
  • mergeMethod: merge | squash | rebase (default: squash)

8p0s · Merge-conflicts lifecycle event

New status + event

// types.ts
type SessionStatus =
  | "mergeable"
  | "merge_conflicts"  // new
  | "merged" | ...

// lifecycle-manager.ts — polling loop
if (!mergeReady.noConflicts) return "merge_conflicts";

// event map
case "merge_conflicts": return "merge.conflicts";

Example config

reactions:
  merge-conflicts:
    auto: true
    action: send-to-agent
    message: >
      Your PR has merge conflicts.
      Run: git fetch origin && git rebase origin/main
      Resolve conflicts, then push.
    retries: 2
Without this event, an agent could only learn about conflicts by polling or if a human intervened. Now it is a first-class lifecycle transition.

4xzz · Hook observer documentation

What hooks actually do

Agent runs gh pr create
→
Command completes
→
PostToolUse hook fires
→
AO metadata updated

The hook cannot modify command behavior, inject flags, or prevent execution. It is an observer that syncs AO session metadata post-command.
