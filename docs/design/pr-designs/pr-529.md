# [agento] feat(skeptic): add Rule 14 (Scope Boundary) + Gate 8d marker

## Overview

**PR:** [#529](https://github.com/jleechanorg/agent-orchestrator/pull/529) · **Status:** Open · **Date:** May 2026

**Author:** @jleechan2015

Adds Rule 14 (Scope Boundary) to the skeptic evaluation prompt, parallel to the established Rule 12 (Goals) and Rule 13 (Tenets) pattern. Rule 14 checks that diff changes stay within the stated scope boundaries declared in the PR description, with a no-scope-claims fallback (14e) and a mechanical-update exception (lockfile/package-lock.json).

Introduces `<!-- skeptic-gate-8d:PASS|FAIL -->` as a diagnostic sub-marker for scope boundary gaps. The Gate 8 PASS template is updated to always include all four sub-markers (8a/8b/8c/8d) as informational output; in FAIL output only relevant sub-markers are emitted.

Also removes the `harness-artifacts` module from `packages/core` — this module implemented long-running harness artifact schemas (research.md, plan.md, handoff.md) that were not wired up to any runtime consumer and are being cleaned up in this PR to reduce dead code.

## Architecture

### Rule 14 Design

Rule 14 follows the established pattern from Rule 12 and Rule 13:

```
Rule 12 — Goals Proof:    PR description Goals section → test validation
Rule 13 — Tenets:         PR description principles   → implementing code
Rule 14 — Scope Boundary: PR description scope claims → diff file coverage
```

Sub-rule breakdown:
- **14a**: Extract scope claims (explicit "Scope:", "In Scope:", "adds X", "modifies Z" language)
- **14b**: Identify claimed file/directory coverage
- **14c**: FAIL if diff touches unmentioned files/dirs not explained by Goals or Tenets
- **14d**: FAIL if PR explicitly scopes to X but diff also changes Y without justification
- **14e**: Fallback — if no scope claims found, do not FAIL; scope ambiguity is informational only

### Gate 8 Sub-marker Contract

| Sub-marker | Condition for FAIL output emission |
|------------|-------------------------------------|
| 8a | Rule 12 gaps (goals proof) |
| 8b | Rule 13 gaps (tenets adherence) |
| 8c | Evidence provenance gaps |
| 8d | Rule 14 gaps (scope boundary) |

In PASS output: all four sub-markers (8a/8b/8c/8d) are always emitted as informational diagnostics. Sub-markers do not independently gate merge — Gate 8 is the merge gate.

### Green Gate Simplification

The Green Gate check-run posting was simplified: the previous implementation looked up any existing check run for the same name+SHA (to PATCH it and avoid 422 duplicate errors) before deciding to POST a new one. This conditional PATCH/POST logic is removed; the workflow now always POSTs a new check run. This works because GitHub allows multiple check runs with the same name for a given SHA and the gate polling logic matches by name/status rather than by check-run ID.

### harness-artifacts Removal

`packages/core/src/harness-artifacts.ts` implemented typed schemas and serialization for long-running harness artifacts (research.md, plan.md, handoff.md) intended for multi-session context resets. The module was never integrated into any runtime caller (`context-reset-manager.ts` was not implemented). Removing the module and its test file reduces dead code and eliminates 1229 lines of unmaintained schema surface.

## Design Invariants

- `hasCompletePassingGateMarkers` enforces gates 1–8 only (unchanged by this PR)
- `extractSkepticGateMarkers` now recognizes 8a/8b/8c/8d for informational display in VERDICT comments
- Rule 14 does not produce a FAIL when no scope claims exist in the PR description (14e fallback)
- Mechanical updates (package-lock.json, lockfile refreshes) are exempt from Rule 14c
