# [agento] fix: run skeptic-cron regardless of backfillAllPRs

## Overview

Fixes two related issues:

1. The `backfillAllPRs !== false` gate in lifecycle-manager.ts was incorrectly skipping skeptic-cron evaluations for repos configured with `backfillAllPRs: false`.
2. The skeptic-gate polling jq filter excluded skeptic VERDICT comments when `SKEPTIC_BOT_AUTHOR == PR_author`, causing gate timeouts for this repo (where the bot author is the PR author).

**PR:** [#497](https://github.com/jleechanorg/agent-orchestrator/pull/497)

**Author:** @jleechan2015

---

## Problem

worldarchitect.ai has `backfillAllPRs: false` in its AO config to disable automatic PR backfill. However, the skeptic-cron call in lifecycle-manager was gated behind `backfillAllPRs !== false`, causing skeptic evaluations to be silently skipped for worldarchitect PRs.

The gate at `lifecycle-manager.ts:2765`:
```typescript
if (skepticProject && skepticProject.backfillAllPRs !== false) {
  void runLocalSkepticCron(...)
}
```

This meant skeptic-cron only ran when `backfillAllPRs` was `true` or `undefined`, but NOT when explicitly set to `false`.

---

## Fix

Remove the `backfillAllPRs !== false` condition from the skeptic-cron guard. The backfill flag only controls worker auto-spawn; skeptic-cron must run independently to service existing open PRs.

**Before:**
```typescript
if (skepticProject && skepticProject.backfillAllPRs !== false) {
```

**After:**
```typescript
if (skepticProject) {
```

---

## Architecture

```text
lifecycle-manager poll loop:
  ├── backfillAllPRs gate → controls worker auto-spawn (session creation for PRs)
  └── skeptic-cron gate → should run independently, always evaluates open PRs
```

**Key insight:** backfill and skeptic-cron are orthogonal concerns:
- backfill: spawns worker sessions for PRs with no active session
- skeptic-cron: evaluates existing PRs for verdict re-use

## Fix 2: jq-filter author-trust predicate (SKEPTIC_BOT_AUTHOR == PR_author)

### Problem

The jq filter author-trust predicate in `skeptic-gate-reusable.yml` and `skeptic-cron-reusable.yml` used:

```jq
((user.login == SKEPTIC_BOT_AUTHOR) and (user.login != PR_AUTHOR))
```

When `SKEPTIC_BOT_AUTHOR == PR_AUTHOR`, the conjunction is always false (author cannot equal AND not equal the same value), causing ALL skeptic VERDICT comments from the bot to be silently dropped.

### Fix

Changed the author-trust logic to accept skeptic bot comments unconditionally when checking for VERDICT comments:

```jq
# Before (always false when bot==PR_author):
((user.login | ascii_downcase) == (SKEPTIC_BOT_AUTHOR | ascii_downcase))
and ((user.login | ascii_downcase) != (PR_AUTHOR | ascii_downcase))

# After (accepts skeptic bot unconditionally):
((user.login | ascii_downcase) == (SKEPTIC_BOT_AUTHOR | ascii_downcase))
```

The fix is applied at 4 polling locations across `skeptic-gate-reusable.yml` and `skeptic-cron-reusable.yml`.

Regression test in `jq-filters.test.ts:642`:

```typescript
it("accepts skeptic VERDICT when SKEPTIC_BOT_AUTHOR equals PR author", () => {
  // Previously the filter rejected this case: (author==bot AND author!=PR_author)
  // always false when bot==PR_author. The fix accepts skeptic bot unconditionally.
  const comments: IssueComment[] = [
    { id: 450, user: { login: "jleechan2015" }, body: boundPassBody(SHA_A) },
  ];
  expect(jqVerdictComment(comments, "jleechan2015", SHA_A, "req-1", "jleechan2015"))
    .toContain("VERDICT: PASS");
});
```
