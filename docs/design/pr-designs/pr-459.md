# Design: PR #459 — Fix Prompt/Branch Decision Drift

## Problem Statement

The prompt builder infers tracker-driven branch guidance from `issueContext` availability, but session-manager selects tracker branches independently. When `generatePrompt()` fails silently (non-fatal exception catch), `issueContext` becomes `undefined` while the branch is still tracker-derived. This causes the prompt to contradict actual branch selection:

- **Reality**: Branch is `feat/INT-1343` (tracker-driven)
- **Prompt says**: "Branch name is auto-generated by the orchestrator"

This drift confuses agents and violates the principle that prompts must be truthful about the environment and decisions made.

## Root Cause Analysis

**Location**: `packages/core/src/prompt-builder.ts`, line 97:
```typescript
const trackerDrivenBranching = Boolean(project.tracker && issueContext);
```

This condition uses `issueContext` as a proxy for "branch will be tracker-driven." But in `session-manager.ts` (line 1057), the actual branch selection uses:
```typescript
} else if (spawnConfig.issueId && plugins.tracker && resolvedIssue) {
  branch = plugins.tracker.branchName(spawnConfig.issueId, project);
```

**The disconnect**: When `plugins.tracker.generatePrompt()` fails (line 1122), `issueContext` is `undefined` via silent catch. But `branch` is already set to tracker-derived on line 1058. The prompt builder then sees `issueContext = undefined` and infers `trackerDrivenBranching = false`, lying to the agent.

## Solution Design

### Approach: Explicit State from Caller

Instead of inferring branch strategy from incomplete context, pass an explicit flag from the call site that already computed the branch decision.

**Key principle**: The caller (session-manager) knows whether the branch will be tracker-derived. Don't make the callee (prompt-builder) guess.

### Changes

1. **PromptBuildConfig interface** (prompt-builder.ts):
   - Add `trackerDrivenBranching?: boolean` field
   - This replaces inference from `issueContext`

2. **buildConfigLayer() function** (prompt-builder.ts):
   - Use explicit `trackerDrivenBranching` flag instead of computing from `issueContext`
   - Decouple prompt guidance from issueContext availability

3. **buildPrompt() call site** (session-manager.ts, line 1134):
   - Pass `trackerDrivenBranching: Boolean(!spawnConfig.branch && spawnConfig.issueId && plugins.tracker && resolvedIssue)`
   - Uses the same condition as the branch selection logic (line 1057)
   - Excludes tracker-driven mode when explicit branch override is provided

4. **Test coverage** (prompt-builder.test.ts):
   - Test explicit `trackerDrivenBranching=true` → tracker branch guidance
   - Test explicit `trackerDrivenBranching=false` → auto-generated guidance (even with issueContext)
   - Test explicit `trackerDrivenBranching=true` with missing issueContext (the bug scenario)
   - All tests use inverse assertions to ensure branch modes are mutually exclusive

## Invariant Preserved

The call site passes the exact condition it used to select the branch. This guarantees:
- If `branch = plugins.tracker.branchName(...)` was selected, prompt says "tracker branch"
- If `branch = auto-generated`, prompt says "auto-generated"
- Never a contradiction, regardless of issueContext resolution status

## Edge Cases Handled

1. **generatePrompt() fails**: `issueContext = undefined`, but explicit `trackerDrivenBranching = true` ensures prompt says "tracker branch" ✓
2. **Explicit branch override**: `spawnConfig.branch` provided, so `trackerDrivenBranching = false`, prompt says "auto-generated" ✓
3. **No tracker plugin**: `plugins.tracker = undefined`, so `trackerDrivenBranching = false` ✓
4. **No issue ID**: `spawnConfig.issueId = undefined`, so `trackerDrivenBranching = false` ✓

## Testing Strategy (TDD Red-Green)

- **RED**: Original code infers from issueContext. Test fails when issueContext is missing but branch is tracker-derived (createGap call).
- **GREEN**: New code uses explicit flag. All edge cases pass via inverse assertions (no ambiguity).

## Risk Assessment

**Low risk**:
- Change is localized to prompt-builder and its call site
- No changes to branch selection logic (already correct)
- No changes to session lifecycle or agent runtime
- Backward compatible: `trackerDrivenBranching` is optional (defaults to false/undefined)

**Mitigation**:
- Regression test validates the exact failure mode (branch-derived without issueContext)
- Inverse assertions prevent future code from accidentally enabling both branch modes simultaneously
- Tests cover all 4 scenarios: explicit true/false, with/without issueContext

## Acceptance Criteria

✅ Prompt guidance matches actual branch selection decision in all scenarios  
✅ No false positives: agent never told to auto-generate when tracker branch is selected  
✅ No false negatives: agent knows to create tracker branch when applicable  
✅ Explicit state passed from caller (not inferred by callee)  
✅ Regression test validates `trackerDrivenBranching=true` with missing issueContext  
✅ All tests pass with inverse assertions ensuring mutual exclusivity
