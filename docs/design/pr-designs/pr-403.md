# [agento] fix(scm-github): allow fetch retry when stale worktree unregistered even if dir exists

## Overview

Fixes a bug in `maybeRemoveStaleCheckedOutWorktree` where the function returned
`false` when the stale worktree directory still existed on disk after being
unregistered from git, preventing the fetch retry from succeeding.

**PR:** [#403](https://github.com/jleechanorg/agent-orchestrator/pull/403) · **Status:** Open · **Date:** April 11, 2026

**Author:** @jleechan2015

---

## Problem

When `checkoutPR` encountered a "refusing to fetch into branch" error caused by
a stale worktree registration, it called `maybeRemoveStaleCheckedOutWorktree` to
unregister the stale entry and retry the fetch. However, the function's return
value included an `existsSync` check:

```typescript
// OLD: returns false if directory still exists on disk
return !remainingWorktrees?.has(resolvedStale) && !existsSync(resolvedStale);
```

Git's "refusing to fetch" error is based on **registered** worktrees, not
directory existence. A directory left behind by a previous AO session does not
prevent git from fetching — only the registration does. So the `existsSync`
check was causing the function to return `false` (i.e., "retry not safe") even
after successfully unregistering the stale entry.

## Fix

Remove the `existsSync` guard from the return condition:

```typescript
// NEW: returns true once unregistered, regardless of directory existence
return !remainingWorktrees?.has(resolvedStale);
```

## Test Coverage

- New test: "retries fetch after git removes a stale worktree entry even if the
  directory still exists" — creates an actual directory on disk, confirms fetch
  is retried and succeeds after unregistration.
- Additional tests for non-AO worktrees (should not be removed) and unregistered
  AO paths (early return path).
