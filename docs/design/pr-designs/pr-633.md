# [agento] [antig] refactor(agent-grok): switch from post-launch to inline prompt delivery

## Overview

**PR:** [PR #633](https://github.com/jleechanorg/agent-orchestrator/pull/633) Â· **Status:** Open Â· **Date:** June 1, 2026

**Author:** @antigravity-agent

This PR refactors prompt delivery in `agent-grok` to be inline (sent at boot time via command-line argument) instead of post-launch. It also hardens branch resolution resilience in `resolveBaseBranch`, prevents environment pollution, gracefully handles `limit=NaN` in the sessions API, and ensures robust WebSocket integration test cleanups.

---

## Problems & Requirements

### 1. Race Conditions in Grok Prompt Delivery
Previously, the agent-grok plugin delivered the task prompt post-launch via terminal input streaming. This created race conditions during session restoration and reconnection:
- The system could not cleanly distinguish between a fresh launch and a resume/reconnect when delivering the prompt.
- Reconnected prompt delivery could cause prompt duplication or shell pollution.

### 2. Base Branch Stale Diffs & Environment Pollution in CI
During wholesome test execution:
- `resolveBaseBranch` prioritized local branch references over remote-tracking references (`origin/*`). In CI or developer checkouts where a local stale branch (e.g. `main`) exists, this resulted in diffing against outdated references, leading to false negatives or missed lint/style violations.
- Direct environment modification of `process.env.GITHUB_BASE_REF` in tests caused pollution because it was not properly restored, risking subsequent test suite failures.
- Module discovery ran git fetch commands at import-time, leading to discover-phase delays in Vitest.

### 3. Sessions API limit=0 and NaN Handling
The `/api/sessions` API endpoint checked the `limit` query parameter using `typeof limit === "number"`. However, in JavaScript, `NaN` has a type of `"number"`, bypassing standard type checks. Additionally, passing `limit=0` resolved to the default page limit of 100 instead of falling back to a minimum valid limit of 1, causing unexpected large payloads at runtime.

### 4. Flaky WebSocket Integration Tests
The direct-terminal-ws integration tests upgrade HTTP requests to WebSocket connections. If a test assertion failed before the connection was explicitly closed, the socket remained orphaned and active, causing the test runner to hang indefinitely in CI environments.

---

## Solutions

### 1. Inline Prompt Delivery in buildGrokCommand
The prompt is now delivered inline directly in the launch command-line args (appended after `--` with proper `shellEscape` formatting) on initial spawn:
- Inline delivery is keyed off `sessionId == null` inside `buildGrokCommand`.
- Explicit reconnect command generation via `getRestoreCommand` omits/suppresses the prompt since the session has already booted.

### 2. Remote-First Base Branch Resolution & Lazy Sourcing
- Prioritized remote-tracking branch validation (`origin/${validated}`) first before bare local references.
- Added a non-destructive fallback `git fetch` operation specifically to fetch the remote-tracking ref if it does not exist locally.
- Restructured branch resolution to perform lazy evaluation at query time rather than top-level module load time.
- Hardened test harness cleanups in `wholesome.test.ts` to guarantee `GITHUB_BASE_REF` is properly restored without environment pollution.

### 3. Typesafe limit validation in Sessions API
Hardened `/api/sessions/route.ts` to validate the pagination limit parameter using:
```typescript
const parsedLimit = limitParam ? parseInt(limitParam, 10) : undefined;
const limit = typeof parsedLimit === "number" && !isNaN(parsedLimit)
  ? Math.max(1, Math.min(parsedLimit, 500))
  : limitParam ? DEFAULT_PAGE_LIMIT : undefined;
```
This ensures invalid limits (`NaN`) fall back to the default `100`, while a limit of `0` correctly maps to the minimum safe limit of `1`.

### 4. Secure afterEach socket cleanups
Integration tests now register all created WebSocket instances with a local `closers` array immediately after instantiation:
```typescript
const ws = new WebSocket(url);
closers.push(ws);
```
An `afterEach` hook runs through the array and terminates every socket instance, preventing dangling sockets and test hangs.
