# Design: PR #461 — Document VERDICT_LINE_RE vs STRICT_VERDICT_RE Separation

## Problem Statement

Two different regex matchers for `VERDICT` tokens exist in the skeptic codebase with different acceptance semantics:

- `VERDICT_LINE_RE` (verdict-utils.ts) — accepts `SKIPPED`
- `STRICT_VERDICT_RE` (llm-eval.ts) — accepts only `PASS|FAIL`

The distinction was not documented, making it unclear why the same token (`SKIPPED`) is accepted by one regex but rejected by the other.

## Context

### `STRICT_VERDICT_RE` — llm-eval.ts (strict internal validation)

**Purpose**: Validates tool output from Codex, Claude, Cursor, and Gemini agents during LLM evaluation.

**Accepted tokens**: `PASS`, `FAIL` only.

```typescript
const STRICT_VERDICT_RE = buildVerdictLineRe(["PASS", "FAIL"]);
```

When `STRICT_VERDICT_RE.test(output)` fails (no match), the chain is fail-closed:

```typescript
if (!STRICT_VERDICT_RE.test(output)) {
  return {
    validVerdict: false,
    error: `Codex output missing VERDICT line...`,
  };
}
```

**Design principle**: SKIPPED is not a valid evaluation outcome in the strict chain. If the model cannot reach a decision, that is treated as a missing verdict → fail-closed. Infrastructure failures must block merges.

### `VERDICT_LINE_RE` — verdict-utils.ts (GitHub comment parsing)

**Purpose**: Parses `VERDICT:` lines emitted by LLM output into GitHub comments during skeptic runs.

**Accepted tokens**: `PASS`, `FAIL`, `SKIPPED`.

```typescript
export const VERDICT_LINE_RE = buildVerdictLineRe(["PASS", "FAIL", "SKIPPED"]);
```

The `SKIPPED` case is handled in `skeptic-gate.yml` polling logic:

```yaml
# skeptic-gate.yml
if [[ "$VERDICT" == "SKIPPED" ]]; then
  echo "VERDICT: SKIPPED — infra issue, not blocking"
  exit 0  # exit cleanly, don't fail the cron gate
fi
```

**Design principle**: SKIPPED means "the LLM could not reach a decision" (not an infra failure). In the GitHub comment / cron polling context, a SKIPPED verdict means the skeptic run did not complete — exiting 0 prevents infra issues from permanently blocking the cron job.

## Why Two Separate Designs

| Dimension | `STRICT_VERDICT_RE` (llm-eval.ts) | `VERDICT_LINE_RE` (verdict-utils.ts) |
|---|---|---|
| Context | Internal LLM eval chain | GitHub comment parsing (skeptic-gate polling) |
| SKIPPED meaning | Missing verdict → fail-closed | Model chose SKIPPED → exit 0 |
| Rejects SKIPPED? | Yes (fail-closed) | No (graceful skip) |
| Infra failure handling | Blocks (fail-closed) | Exits 0 (doesn't block cron) |

These are intentionally separate concerns:

1. **Fail-closed for evaluation**: `llm-eval.ts` drives actual merge decisions. A SKIPPED verdict from the model means the evaluator couldn't do its job → must not approve.
2. **Graceful skip for polling**: `skeptic-gate.yml` polls GitHub for verdicts posted by the lifecycle worker. If the LLM emits SKIPPED, the cron should exit cleanly rather than block indefinitely on infra issues.

## Solution

Add JSDoc comment to `VERDICT_LINE_RE` in `verdict-utils.ts` clarifying the SKIPPED acceptance and its relationship to `STRICT_VERDICT_RE`.

**No behavioral change** — all logic remains identical.

## Risk Assessment

**Negligible**: Pure documentation change. No code logic modified.

- All existing tests (156 skeptic, 47 llm-eval, 96 core skeptic) pass
- No changes to verdict parsing or evaluation behavior
- No impact on merge gate, skeptic-cron, or skeptic-gate workflows
