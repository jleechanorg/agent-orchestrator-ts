# [agento] feat(skeptic-cron): bounded concurrency

## Overview

**PR:** [#524](https://github.com/jleechanorg/agent-orchestrator/pull/524) · **Status:** Open · **Date:** May 5, 2026

**Author:** @jleechan2015

This PR changes local skeptic cron execution from one-at-a-time PR evaluation to bounded batch evaluation. The default cap is 3 concurrent skeptic verifications per project, with a configurable `maxConcurrentSkepticReviews` override for callers that need a different cap.

---

## Problem

`runLocalSkepticCron` evaluated eligible open PRs sequentially. That made one slow `ao skeptic verify` call delay every later PR in the same project, even when the later PRs were independent.

The concurrency change needed to preserve existing safety behavior:

1. Per-project throttle and pending guards must still prevent overlapping cron cycles.
2. SHA dedup must still skip already-evaluated PR heads.
3. One failed PR evaluation must not cancel other PRs in the batch.
4. Observer telemetry failures must not change skeptic evaluation results.
5. Bad concurrency configuration must not produce infinite loops or unbounded execution.

---

## Fix

The implementation adds `maxConcurrentSkepticReviews` to `SkepticCronParams`, normalizes it to a positive finite integer, and defaults to 3.

Eligible non-draft PRs are evaluated in bounded slices:

```text
eligible PRs -> chunks of maxConcurrent -> Promise.allSettled(chunk.map(evaluateOnePR))
```

`evaluateOnePR` owns the per-PR flow:

- reuse an active PR session or create a synthetic session
- read and compare the current PR head SHA
- skip if the same project/PR head already has a completed evaluation
- run `runSkepticReview(session, { postComment: true })`
- record telemetry on best effort only
- cache the SHA after successful review execution

The code uses `Promise.allSettled` so one rejected PR evaluation does not reject the whole batch. Observer writes in the success path are wrapped independently, so telemetry exceptions do not prevent SHA caching or flip a successful evaluation to `false`.

---

## Concurrency Contract

| Input | Behavior |
|-------|----------|
| omitted | default cap 3 |
| positive integer | use the provided value |
| positive fraction | truncate to integer |
| `0`, negative, `NaN`, `Infinity` | fallback to default cap 3 |

The cap applies per invocation and per project. It does not spawn extra lifecycle workers by itself; it bounds how many independent PR skeptic reviews the local cron loop starts in parallel for one project.

---

## Risk And Mitigation

| Risk | Mitigation |
|------|------------|
| Higher verifier load | conservative default cap of 3 |
| One PR failure cancels a batch | `Promise.allSettled` |
| Observer failure suppresses valid completion | success-path observer calls are individually caught |
| Invalid cap causes infinite loop | normalization rejects non-finite and non-positive values |
| Brittle concurrency tests | tests assert observed in-flight count instead of wall-clock timing |

## Test Coverage

TypeScript tests in `packages/core/src/__tests__/skeptic-cron-local.test.ts` cover:

| Test area | Validates |
|-----------|-----------|
| bounded concurrency | max observed in-flight reviews never exceeds configured cap |
| default cap | omitted config defaults to 3 |
| sequential override | cap 1 keeps max in-flight at 1 |
| invalid caps | `NaN`, `Infinity`, negatives, zero, and fractions are normalized safely |
| batch isolation | one rejected PR does not block later PRs |
| observer isolation | observer throw does not cancel other PRs or skip SHA caching |
| existing behavior | throttle, pending guard, synthetic sessions, SHA dedup, and fail-open SHA lookup |
