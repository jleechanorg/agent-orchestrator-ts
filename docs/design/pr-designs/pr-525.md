# [agento] fix(launchd): inject CLAUDE_BINARY into lifecycle worker plist env

## Overview

Ensure `tryClaudePrint()` succeeds when `CLAUDE_BINARY` env var is set in the launchd worker environment.

Without this fix, lifecycle workers lack `CLAUDE_BINARY` in their env, causing `tryClaudePrint()` to skip the preferred CLI binary and fall through to GUI app candidates that ETIMEDOUT in headless launchd context.

**PR:** [#525](https://github.com/jleechanorg/agent-orchestrator/pull/525) · **Status:** Open · **Date:** May 2026

**Author:** @jleechan2015

---

## Architecture

**Problem:** `tryClaudePrint()` iterates `CLAUDE_BINARY_CANDIDATES` in order. Without `CLAUDE_BINARY` set in the launchd plist `EnvironmentVariables`, lifecycle workers fall through to GUI app candidates that ETIMEDOUT.

**Precedence:** `CLAUDE_BINARY` (highest) > `CLAUDE_BINARY_PATH` > `$HOME/.local/bin/claude` (default)

## Claim Class

`infra` — launchd configuration bug fix; no separate design doc or bead required.

## Design Invariants

- `setup-launchd.sh` renders the plist with the calling env's `CLAUDE_BINARY` (or `CLAUDE_BINARY_PATH`, or default) at install time
- The plist injects `CLAUDE_BINARY` into the launchd worker's initial environment; this is the mechanism fixed by this PR
- Child processes spawned from the worker inherit `CLAUDE_BINARY` from the worker env
