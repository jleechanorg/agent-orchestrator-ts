# [agento] feat: source API keys from shell init files at daemon startup (bd-g884)

## Overview

**PR:** [#528](https://github.com/jleechanorg/agent-orchestrator/pull/528) ┬Ę **Status:** Open ┬Ę **Date:** May 7, 2026

**Author:** @jleechan2015

This PR adds configurable `envSource` support to the AO config, allowing the daemon to source API keys (MINIMAX_API_KEY, ANTHROPIC_API_KEY, etc.) from shell init files at startup. This solves the root cause: the AO daemon starts without a login shell and never inherits env vars defined in `~/.bashrc`.

---

## Problem

The AO daemon does NOT inherit from `~/.bashrc` when it starts ŌĆö it launches as a non-login, non-interactive process. The MiniMax plugin (and other plugins) read `process.env.MINIMAX_API_KEY`, which is always `undefined` in worker sessions unless the key was exported to the launchd environment.

Users who define `export MINIMAX_API_KEY=...` in their `~/.bashrc` (or `~/.zshrc`, `~/.profile`) have no way to expose it to AO worker sessions without duplicating secrets into the YAML config file.

---

## Solution

### envSource config field

A new `envSource` field in `OrchestratorConfig` (top-level or under `defaults`) accepts a list of shell init file paths to source at config-load time:

```yaml
defaults:
  runtime: tmux
  agent: claude-code
  envSource:
    - ~/.bashrc
```

Default: `["~/.bashrc"]`.

### sourceEnvFile() ŌĆö shell sourcing utility

The new `applyEnvSource(files)` function in `packages/core/src/env-source.ts` runs for each configured file:

```bash
bash --noprofile --norc -i -c 'source "$1" > /dev/null 2>&1; env' -- <expanded-path>
```

Three shell flags are intentional:
- `--noprofile` ŌĆö do not source `/etc/profile` or login scripts before interactive startup
- `--norc` ŌĆö do not source `~/.bashrc` implicitly before the explicit `source "$1"`; prevents implicit startup-file leakage where bash sources `~/.bashrc` automatically before the configured file exports vars
- `-i` ŌĆö interactive mode so that `[[ $- != *i* ]] && return` guards in common `.bashrc` templates do not skip the API-key exports below them

A prefix-matched diff against a pre-source `process.env` snapshot ensures only new/changed vars are merged, preventing PATH/PS1 pollution.

### Security: path allowlist

`envSource` schema validation rejects any path that is not a shell dotfile under `~` or `/etc/environment`. The regex:

```regex
^~\\/\\.[a-zA-Z][a-zA-Z0-9_-]*$
```

Requires a literal dot after `~/`, which blocks:
- `/tmp/evil.sh` (absolute path)
- `~/scripts/env.sh` (non-dotfile under home)
- `~/../otheruser/evil.sh` (path that escapes `$HOME`)
- `~/bashrc` (unescaped dot ŌĆö `.bashrc` requires the dot)

### Allowed prefixes

Only env vars with known API-key prefixes are imported:

```typescript
const ALLOWED_PREFIXES = [
  "MINIMAX_",
  "ANTHROPIC_",
  "OPENAI_",
  "MCP_AGENT_MAIL_",
  "AO_",
] as const;
```

### Shell invocation: `--noprofile --norc -i`

Three flags on `bash` are intentional:

| Flag | Purpose |
|------|---------|
| `--noprofile` | Skip login profile scripts (`/etc/profile`, `~/.bash_profile`) |
| `--norc` | Skip `~/.bashrc` implicit sourcing ŌĆö prevents startup-file leakage where bash sources `~/.bashrc` automatically before the configured file |
| `-i` | Interactive mode ŌĆö so that `[[ $- != *i* ]] && return` guards in common `.bashrc` templates do NOT skip API-key exports below them |

### Shell separator: `;` not `&&`

The sourcing command uses `;` (semicolon) not `&&`:

```bash
source "$1" > /dev/null 2>&1; env
```

`&&` short-circuits ŌĆö if the sourced file exits non-zero (e.g. `set -e`, a failing command), `env` never runs and no vars are captured. `;` always runs `env` regardless of exit code.

## Compatibility Notes

### Interactive shell flag (`-i`)

Using `bash -i -c` means the sourced file runs as if in an interactive shell. This is intentional:
- Common `.bashrc` templates guard exports with `[[ $- != *i* ]] && return` ŌĆö without `-i` those exports are skipped.
- `--noprofile` prevents bash from sourcing `/etc/profile` or `~/.bash_profile` during startup, so those login scripts are never loaded (the `-i` flag does not re-enable them).
- `--norc` prevents bash from sourcing `~/.bashrc` implicitly on startup, so the explicit `source "$1"` call is the only sourcing action (no duplicate-loading risk from implicit startup-file leakage).
- The `[[ $- != *i* ]] && return` guard in common `.bashrc` templates is safe to bypass with `-i`: if a user's `.bashrc` uses this guard, `-i` makes `$-` contain `i`, so the guard evaluates false and exports run. If the guard is not present, `-i` has no effect and exports run as normal.

### File order and override

`applyEnvSource` calls `sourceEnvFile` for each file in sequence, passing a fresh `process.env` snapshot before each file. Later files override earlier ones for the same key.

### Non-interactive-only setups

Users who run commands in a completely non-interactive context (no TTY, no `bash -i`) should use `/etc/environment` (the only absolute path allowed by schema). `/etc/environment` is a plain `KEY=VALUE` file ŌĆö **no `export` keyword required or supported**.

**Important**: `/etc/environment` is parsed by direct file read + line split, NOT through bash sourcing. Entries must use the bare `KEY=VALUE` form. Lines using `export KEY=VALUE` are ignored (the key would be parsed as the literal string `"export"`, which matches no prefix). This differs from shell dotfiles which require `export` for vars to appear in `env` output.

## Risk And Mitigation

| Risk | Mitigation |
|------|------------|
| `-i` flag causes bash to hang or wait for input | `stdin: pipe` in execFileSync + `> /dev/null` redirect |
| `~/.bashrc` sources `~/.bash_profile` ŌåÆ infinite loop | execFileSync timeout (10s) catches runaway loops |
| Malicious envSource config path | schema `.refine()` allowlist blocks all absolute paths except `/etc/environment` |
| API key leaked in process.env snapshot | allowed-prefix filter prevents non-API vars from being merged |
| Daemon startup blocked by bad envSource | try/catch in `sourceEnvFile` ŌĆö failure is silent, no throw |
